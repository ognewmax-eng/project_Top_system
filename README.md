# Система трудовых отрядов

Веб-приложение для регистрации участников программы «Трудовое лето»: подача заявок, загрузка документов, личный кабинет и панель администратора.

**Подробная инструкция по выкладке на хостинг Reg.ru (без VPS):** см. [DEPLOY_REG_RU.md](DEPLOY_REG_RU.md).

---

## Стек технологий

| Категория      | Технологии |
|----------------|------------|
| **Frontend**   | React 18, TypeScript, Vite 6, Tailwind CSS 4 |
| **Backend**    | PHP (скрипты в `public/api/` — при сборке копируются в `dist/api/`) |
| **Хранение**   | Клиент: localStorage (заявки, пользователь); сервер: JSON (`api/data/`) при наличии PHP; файлы — **Яндекс.Диск** (папка `top` → смены → ФИО) или локально `api/uploads/` |

---

## Требования

- **Node.js** 18+ (рекомендуется 20+)
- **npm** или **pnpm**
- Для деплоя: **PHP** 7.4+ на хостинге (например, Reg.ru Shared)

## Локальная разработка

### 1. Установка зависимостей

```bash
npm install
```

или с pnpm:

```bash
pnpm install
```

### 2. Локальный запуск (режим разработки)

Запуск dev-сервера с горячей перезагрузкой:

```bash
npm run dev
```

Откройте в браузере адрес, который выведет Vite (обычно **http://localhost:5173**). Изменения в коде подхватываются автоматически.

**Примечание:** В режиме разработки формы и загрузка файлов обращаются к `/api/`. Чтобы они работали локально, нужен PHP: либо поднимите отдельно сервер из папки `dist` после сборки (`npm run build` и затем `php -S localhost:8080 -t dist`), либо настройте прокси в `vite.config.ts` на ваш PHP-сервер. Без PHP заявки сохраняются только в localStorage (резервный режим).

### 3. Сборка проекта

```bash
npm run build
```

В корне проекта появится папка **`dist`** с фронтендом и папкой **`api`** (содержимое `public/api/` копируется туда автоматически).

### 4. Просмотр собранной версии локально

После `npm run build` можно раздать `dist` статическим сервером:

```bash
npx serve dist
```

Чтобы проверить работу форм и загрузки файлов, запустите из корня проекта встроенный PHP-сервер, указав папку `dist` как корень:

```bash
php -S localhost:8080 -t dist
```

Откройте в браузере **http://localhost:8080**.

## Деплой на Shared-хостинг (Reg.ru)

Подробная пошаговая инструкция (ZIP/FTP, права, Яндекс.Диск, проверка) — в отдельном файле **[DEPLOY_REG_RU.md](DEPLOY_REG_RU.md)**. Кратко:

### Шаг 1. Сборка

На своём компьютере в корне проекта выполните:

```bash
npm install
npm run build
```

Убедитесь, что в корне появилась папка **`dist`**.

### Шаг 2. Содержимое для загрузки

В хостинг нужно загрузить **всё содержимое** папки `dist`, а не саму папку `dist`:

- `index.html` — в **корень** сайта (например, `public_html` или `www`)
- папка `assets/` — в корень сайта
- папка `api/` — в корень сайта (внутри неё будут `upload.php`, `save_data.php`, `uploads/` и т.д.)

В итоге на хостинге структура должна быть такой:

```
public_html/          (или www/, htdocs/ — как у вашего хостинга)
├── index.html
├── assets/
│   ├── index-xxxxx.js
│   └── index-xxxxx.css
├── api/
│   ├── upload.php
│   ├── upload_to_yandex.php
│   ├── get_yandex_file.php
│   ├── save_data.php
│   ├── uploads/       (пустая или с .htaccess)
│   └── data/         (создаётся скриптами для JSON)
```

### Шаг 3. Загрузка файлов на хостинг

1. Откройте панель управления хостингом Reg.ru (например, cPanel или «Файловый менеджер»).
2. Перейдите в корневую папку сайта (`public_html`, `www` или аналог).
3. Загрузите всё содержимое `dist`:
   - можно архивировать содержимое `dist` в ZIP, загрузить ZIP на хостинг и распаковать;
   - или подключиться по FTP/SFTP и скопировать файлы и папки в корень сайта.

### Шаг 4. Права на папки

Убедитесь, что папки, в которые PHP пишет файлы, доступны для записи веб-серверу (обычно права **755** для папок, **644** для файлов):

- `api/uploads/` — для загружаемых файлов;
- `api/data/` — для JSON с заявками (если скрипты создают её автоматически, при необходимости создайте папку вручную и выставите права 755).

В файловом менеджере Reg.ru обычно есть «Права доступа» / «Chmod» для папок и файлов.

### Шаг 5. Проверка

1. Откройте в браузере адрес вашего домена — должна открыться главная страница приложения.
2. Заполните форму регистрации и отправьте заявку — данные должны сохраняться через `api/save_data.php`.
3. При загрузке документов проверьте, что файлы появляются в `api/uploads/` и что в `api/uploads/` действует запрет на выполнение скриптов (см. раздел «Безопасность» ниже).

## Загрузка файлов на Яндекс.Диск

Документы участников по умолчанию загружаются на **Яндекс.Диск** в папку `top` со структурой:

- `top/1 смена/` — файлы участников 1-й смены  
- `top/2 смена/` — 2-й смены  
- `top/3 смена/` — 3-й смены  

Внутри каждой смены создаются подпапки по **ФИО** участника, в них сохраняются загруженные файлы.

**Настройка:**

1. Зарегистрируйте приложение в [Яндекс OAuth](https://oauth.yandex.com/) и получите OAuth-токен с правами на запись в Облако (`cloud_api:disk.write`).
2. На сервере задайте переменную окружения **`YANDEX_OAUTH_TOKEN`** (токен) для PHP,  
   **или** создайте в папке `api/` файл **`.yandex_oauth_token`** (одна строка — токен). Этот файл добавлен в `.gitignore`, в репозиторий не попадает.
3. После деплоя скрипты `upload_to_yandex.php` и `get_yandex_file.php` должны лежать в `api/` рядом с остальными PHP.

Без токена загрузка на Диск недоступна; форма при этом может сохранять заявку без вложений (или использовать локальный `upload.php` при отключённой интеграции с Яндексом).

## Безопасность

- В `api/uploads/` размещён `.htaccess`, запрещающий выполнение скриптов (PHP и др.) в этой папке.
- В `upload.php` и `upload_to_yandex.php` выполняется проверка расширений и размера файлов; при необходимости список разрешённых типов можно сузить в коде.
- Пароли в анкетах передаются по HTTPS; в списке заявок (localStorage и при сохранении на сервер) пароль не сохраняется в объекте заявки.
- На стороне сервера при развитии проекта рекомендуется хранить только хэши паролей; OAuth-токен Яндекс.Диска хранить в переменной окружения или в файле вне веб-корня с правами 600.

## Особенности работы данных

- Заявки при успешной отправке сохраняются в **localStorage** (`top_applications`, `top_user`) и при наличии PHP — в `api/data/applications.json`. Админ-панель и счётчики смен читают данные из localStorage; при очистке хранилища или с другого устройства список заявок будет пуст, если не настроена синхронизация с серверным JSON.
- Пароль в объекте заявки в список не попадает (не сохраняется в `top_applications`).

## Возможное развитие

- **Синхронизация:** загрузка списка заявок с сервера (из `api/data/applications.json`) при входе в админку или при открытии сайта.
- **S3 (или аналог) для файлов:** в `api/upload.php` помечены комментарии для замены сохранения на вызов облачного хранилища.
- **База данных:** при переносе на VPS/выделенный сервер — заменить запись в JSON на MySQL/PostgreSQL в `save_data.php`.
